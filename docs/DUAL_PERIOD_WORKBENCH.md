# 期货策略与桌面工作台使用说明

## 先说结论

项目现在支持五种策略切换、单品种回测、共享账户多品种回测、训练/验证/最终测试隔离的参数优化，以及零额外 UI 依赖的桌面工作台。当前默认策略仍是 `dual_ma_pullback`；新增 `adaptive_trend_v2` 作为研究推荐，不能仅凭合成数据自动替换默认策略。

工作台包含：

- 可交易品种多选；开仓可选固定手数或按权益自动计算整数手数。
- 内置 69 个国内主流连续合约，可按代码、中文名和交易所板块筛选。
- 可视化持仓、历史成交、逐笔已实现盈亏。
- 共享账户总收益曲线。
- 策略和风控参数调整。
- 网格优化和样本外结果表。

重要边界：仓库中的 `data/domestic_15m` 是确定性合成数据，只用于证明程序链路能运行。它不是近三年真实行情，任何基于它的收益、排名或参数都不能用于实盘决策。

## 一、启动工作台

在项目根目录打开 PowerShell：

```powershell
$env:PYTHONPATH = "src"
python -m futures_quant dashboard
```

如果系统的 `python` 没有指向已安装 pandas/numpy 的解释器，请先创建虚拟环境并安装：

```powershell
py -3.12 -m venv .venv
.\.venv\Scripts\python.exe -m pip install -e .
$env:PYTHONPATH = "src"
.\.venv\Scripts\python.exe -m futures_quant dashboard
```

工作台操作顺序：

1. 在“回测与策略”选择品种池、15 分钟 CSV 目录和文件后缀。
2. “源数据”选择 CSV 本身的周期；内置数据为15分钟，不要改成60分钟。
3. “策略运行周期”可选择15、30、60、120或240分钟；本次 `dual_ma_pullback` 明确定义为15分钟，工作台会自动切回15分钟并在运行前校验。其他策略仍可使用聚合周期。
4. 按住 `Ctrl` 多选品种。共享账户最多允许 5 个品种同时持仓，但研究池可以超过 5 个。
5. 选择策略，在“核心信号、退出管理、双周期参数、自适应参数、账户风控”页签调整参数。
6. 点击“运行共享账户回测”。结果完成后自动打开完整摘要页。
7. 在持仓、成交、盈亏、总收益曲线标签查看结果。
8. 在“参数优化”修改 JSON 网格，点击“开始样本外参数优化”。优化使用当前选定的运行周期。

均线参数的单位是K线根数。本策略的MA13是13根15分钟K线，约3小时15分钟交易时间；MA169是169根15分钟K线，约42小时15分钟交易时间，对应约5～11个交易日，取决于品种是否包含夜盘。

聚合器按品种和连续交易时段分桶，遇到午休、夜盘结束或其他大于一个源周期的间隔会先结束当前桶，不会跨休市间隔拼接K线。每个时段末尾不足完整60分钟的K线会保留，避免丢失午盘和收盘行情。

每次 UI 回测会写入 `reports/workbench/<运行时间>/`。用户可见文件统一使用中文文件名和中文表头，包括 `运行配置.json`、`回测摘要.csv`、`账户权益曲线.csv`、`可视化持仓.csv`、`历史成交.csv`、`历史盈亏.csv`、`拒单记录.csv`。摘要会记录源周期、运行周期、源K线数量和聚合后K线数量；`运行配置.json` 保存品种、策略和风控参数快照，便于复现。

### 一键扫描全部品种

左侧“一键扫描全部69个品种收益”会忽略当前多选状态，使用当前策略、运行周期和风控参数，对品种池中的每个品种分别建立一个相同初始资金的独立账户回测。这样得到的是可比较的单品种研究排名，不受共享账户“最多5个持仓”和不同品种争抢保证金的顺序影响。

扫描结果在“全品种扫描”页按总收益率排序，同时显示年化收益、最大回撤、Sharpe、Calmar、成交数、拒单数、最高保证金占用和错误信息。单个品种缺少行情或合约参数时只标记该行失败，不中断其余品种；扫描过程中可以停止，已有结果仍会保存。

扫描输出位于 `reports/full_universe_scan/<运行时间>/`：

- `全品种收益排名.csv`：中文字段的完整排名。
- `scan_results.csv`：程序可继续读取的稳定英文字段版本。
- `扫描配置.json`：本次策略、周期和风控参数快照。

“选择排名前5品种作为回测池”只负责把前5名放回左侧选择列表，之后仍需运行共享账户回测，重新检验品种相关性、保证金竞争和最多5个持仓约束。不能把同一段历史数据中选出的前5名直接视为未来最优组合；正式研究应在训练期排名，再用未参与排名的样本外区间确认。

## 二、默认策略规则

### 多头入场

1. 上一根15分钟收盘价小于或等于上一根MA169。
2. 当前15分钟最终收盘价大于当前MA169，形成一次明确的向上穿越事件。
3. 信号在当前收盘确认，下一根15分钟K线开盘建立多单；价格持续位于MA169上方不会重复开仓。
4. `order_size > 0` 时使用固定手数；`order_size = 0` 时按权益自动计算。
5. 自动手数向下取整，并取60%单品种保证金预算、单笔止损风险、名义敞口和可选最大手数四者中的最小值；不足1手不交易。

### 多头退出

1. 收盘价小于 `MA169 - 0.1 × ATR(14)` 时发出全部退出信号，下一根开盘平仓。
2. 首次由上向下穿越MA13时，下一根开盘退出当前剩余仓位的30%。
3. 数量按手向下取整且每次至少退出1手；价格必须先重新回到MA13上方，之后再次向下穿越才会继续减仓。
4. 第二次及后续有效穿越继续退出当时剩余仓位的30%，直至清仓。
5. 同一根收盘K线同时满足MA169失效和MA13减仓时，MA169全部退出优先。

### 空头

所有条件镜像：上一收盘不低于上一MA169、当前收盘跌破当前MA169后下一根开盘做空；收盘高于 `MA169 + 0.1 × ATR(14)` 全退；每次由下向上穿越MA13时退出剩余空仓的30%。

### 成交与状态

- 入场、MA169退出和MA13分批退出都由收盘确认，在下一根开盘成交。
- 开仓被保证金、单笔风险或组合风控拒绝后，策略立即同步实际空仓，不会把未成交订单当作持仓管理。
- 分批成交后策略使用实际剩余手数继续计算下一次30%退出。

### 防止未来函数

- 收盘产生的普通信号统一在下一根 K 线开盘成交。
- 所有本策略信号都只用已完成K线，并在下一根开盘成交。
- 参数只用训练期和验证期选择；最终测试期不参与排名。

回测只有 15 分钟 OHLC，无法知道同一根 K 线内的真实价格路径，因此冲突时使用止损优先的保守假设。接入柜台后应把初始止损同步为交易所或柜台条件单，并继续由策略管理均线和 ATR 退出。

## 三、命令行切换策略

策略由 JSON 中的 `strategy.name` 切换：

- `dual_period_reversal`：本次双周期反转策略。
- `dual_ma_pullback`：15分钟MA169穿越入场、MA13递减退出策略（工作台默认；保留旧策略ID以兼容历史配置和报告）。
- `dual_ma`：原双均线基准。
- `adaptive_trend`：原自适应趋势策略。
- `adaptive_trend_v2`：强化自适应研究策略，包含1～5手动态仓位、ATR硬止损、1R保本、2R分批和ATR跟踪。

新策略配置见 `configs/backtest_ma_pullback.json`。单品种回测：

```powershell
$env:PYTHONPATH = "src"
python -m futures_quant backtest --config configs/backtest_ma_pullback.json
```

其他策略在命令行切换周期时修改配置中的数据段：

```json
"data": {
  "symbol": "RB0",
  "path": "data/domestic_15m/RB0_15m.csv",
  "source_interval_minutes": 15,
  "bar_interval_minutes": 60
}
```

共享账户回测：

```powershell
python -m futures_quant portfolio-backtest `
  --config configs/backtest_ma_pullback.json `
  --universe configs/universe_domestic_3y.json `
  --history-dir data/domestic_15m `
  --suffix _15m.csv `
  --symbols RB0,I0,CU0,AU0,M0,TA0,IF0 `
  --output-dir reports/ma_pullback_portfolio
```

生成工程测试用 15 分钟合成数据：

```powershell
python -m futures_quant generate-demo-intraday `
  --universe configs/universe_domestic_3y.json `
  --output-dir data/domestic_15m
```

## 四、使用真实近三年数据

每个品种准备一个文件，例如 `RB0_15m.csv`，字段必须为：

```text
datetime,symbol,open,high,low,close,volume,open_interest
```

要求：

- `datetime` 是 15 分钟 K 线结束时间，统一时区且严格升序。
- 夜盘交易日必须按期货交易日正确归属；不能简单按自然日切割。当前内置聚合器适合无夜盘工程数据，真实夜盘接入时应由数据 Provider 先提供 `trading_day` 映射或把夜盘时间规范化到所属交易日。
- 主连数据必须采用可审计的换月规则和复权方法，并保留原始合约映射，避免幸存者偏差和虚假跳空。
- `RB0` 等主连代码只用于研究，实盘下单前必须映射为当前可交易合约，例如 `RB2610`。
- 合约乘数、tick、保证金率和手续费维护在 `configs/contracts.csv`。

数据来自券商或数据平台 HTTP 接口时，可复制 `configs/http_history.example.json`，把字段映射到上述标准列。对于三年 15 分钟数据，优先使用券商历史行情、天勤/掘金/Tushare 专业数据或交易所授权数据；公开免费接口通常没有完整三年分钟历史。

## 五、参数优化

强化自适应提供独立的15分钟和60分钟两阶段配置。第一阶段只搜索入场、退出、趋势和动量结构；第二阶段继承结构赢家，再搜索波动率目标、ATR止损和跟踪参数。两阶段都只读取训练/验证数据，参数全部冻结后才访问最终20%。

同时运行15分钟和60分钟：

```powershell
python -m futures_quant optimize-adaptive-v2 `
  --universe configs/universe_domestic_3y.json `
  --history-dir data/domestic_15m `
  --suffix _15m.csv `
  --output-dir reports/optimization_adaptive_v2
```

主要新增输出为 `stage1_structure_ranking.csv`、`stage2_risk_ranking.csv`、`strategy_comparison.csv`、`promotion_decision.json` 和跨周期 `timeframe_comparison.csv`。`promotion_decision.json` 永远不会自动修改工作台默认策略；真实行情验证通过后仍需人工确认。

```powershell
python -m futures_quant optimize-strategy `
  --config configs/backtest_ma_pullback.json `
  --optimization-config configs/optimization_ma_pullback.json `
  --universe configs/universe_domestic_3y.json `
  --history-dir data/domestic_15m `
  --suffix _15m.csv `
  --output-dir reports/optimization_ma_pullback
```

不要以“总收益最大”作为唯一目标。执行性更高的选择顺序是：

1. 训练和验证期都为正，且两阶段得分差距不过大。
2. 样本外最大回撤、Sharpe、Calmar 和成交覆盖率达标。
3. 手续费 1.5～2 倍、滑点 2～3 tick 压力下仍不过度恶化。
4. 参数邻域表现平滑，不能只有一个孤立参数点盈利。
5. 最终用共享账户重跑，确认保证金、单笔风险和最多5品种限制没有导致大量信号失效。

不存在可以事先保证的“最优盈利参数”。可执行结论必须在用户提供的真实三年 15 分钟数据上，完成上述样本外与成本压力测试后才能给出。

当前内置合成数据的69品种独立账户工程扫描（2023-01-03至2026-06-30）全部为正，平均收益约768.86%、平均最大回撤约-32.81%，包含48643笔成交动作和6362笔风控拒单。这个结果明显受到合成数据生成方式和60%高仓位放大的影响，只能证明穿越、递减退出、动态手数和风控链路可以运行，不能作为真实收益证据。

## 六、风控约束

- `order_size=0` 时按权益自动定仓；填写正整数时使用固定手数，但仍受所有风险上限约束。
- 本策略默认把单品种保证金预算上限设为当前权益60%，用于复现本次压力测试设想。准备仿真或实盘时建议先降至10%～20%。
- 同时持仓品种不得超过 5 个。
- 工作台本策略配置的单笔风险上限为3%，它仍可能比60%保证金预算更早限制手数；仿真前建议降低到0.5%～1%。
- 同一板块默认最多同时持有 2 个品种，降低黑色、化工等高度相关品种的风险叠加。
- 账户总保证金占用默认不超过 60%。
- 账户单日亏损达到 2% 后禁止新增风险，但允许平仓。
- 单品种名义敞口、保证金和止损风险分别检查，自动手数主动取三者最小值，券商/回测账户层再做一次硬校验。
- 所有拒单写入 `rejections.csv`，不能只看成交而忽略被风控过滤的信号。

## 七、接入国内券商 API

项目边界位于 `src/futures_quant/api/base.py` 和 `src/futures_quant/api/ctp_gateway.py`。策略不直接导入券商 SDK。

推荐顺序：

1. 确认期货公司提供 CTP 仿真或 SimNow，取得交易前置、行情前置、BrokerID、AppID、AuthCode 和 Python SDK。
2. 在 `CtpGateway` 中适配认证、登录、结算确认、行情订阅、报单、撤单和回报回调。
3. 把 tick 聚合成项目标准 15 分钟 `Bar`，并正确处理夜盘交易日。
4. 将主连研究信号映射到实际主力/次主力合约，下单前检查临近交割月、涨跌停、可用资金和交易所限仓。
5. 先只连行情并落盘；用相同数据回放验证策略。
6. 再连接仿真交易，核对持仓、成交、手续费、保证金和断线重连。
7. 实盘前增加人工总开关、单日最大委托次数、撤单频率、行情超时、重复报单幂等和本地状态恢复。

真实密码不要放入 JSON 或仓库，应通过环境变量或系统密钥存储读取。不同期货公司的 CTP SDK 包名和回调字段不同；在没有具体 SDK、仿真地址和账户材料时，仓库只能保留适配边界，不能安全地伪造一个“已接通”的实盘实现。
